   - Проблема осталась. Есть частичный кэш роли, но не устранена первопричина поведения браузера/жизненного цикла страницы.

4. В CRM нет отдельного рабочего варианта “AFM Digital (мы сами)”  
   - Сейчас выбор строится по клиентам из таблицы clients, отдельного “внутреннего CRM пространства агентства” нет.

5. Доступы к AFM Internal / AdminScale / CRM “по галочкам” не доведены

     - нет связки “админ по умолчанию, остальным только при выданных разрешениях”.

6. “View as role” реализован частично  
   - В хедере переключение есть, но многие страницы читают agencyRole, а не effectiveRole, значит симуляция не везде корректна.

7. Переводы не синхронизированы по всем страницам  
   - В новых/доработанных экранах осталось много захардкоженных строк (особенно RU), без полного покрытия i18n.

8. Мониторинг “кто онлайн / последний визит / полезная история действий” не доведен до рабочего админ-инструмента  
   - Таблицы и базовый хук присутствия есть, но:
     - нет полноценного админ-экрана онлайн-статусов;
     - user_activity_log почти не заполняется значимыми событиями;
     - нет удобной связки “пользователь -> значимые действия”.

9. AdminScale перенесен не полностью как продукт  
   - Есть редактор/справочник/экспорт, но нет полной parity по сценариям импорта, совместной работе и постоянному правому помощнику в ключевых режимах так, как ты ожидаешь от оригинала.

## 2) Почему сайт может “прогружаться заново” при возврате

Ключевые причины, которые будем закрывать:
- состояние вкладки может уничтожаться браузером (особенно mobile);
- текущий lifecycle presence-хука инициализируется агрессивно;
- не все проверки инициализации оптимизированы под восстановление из кэша;
- deep init части модулей запускается как “холодный старт”.

Важно: полностью запретить браузеру выгружать вкладку нельзя, но можно сделать возврат практически мгновенным и без видимого “полного перезапуска”.

## 3) План доработки (как именно буду менять)

### Этап A. Стабильность вкладок и возврата (приоритет 1)
1. Переписать lifecycle присутствия (usePresence):
   - убрать блокирующие сценарии unload/beforeunload;
   - перейти на более безопасный page visibility/pagehide паттерн;
   - отправка статуса без ломания bfcache.
2. Усилить восстановление состояния:
   - кэш “последнего стабильного shell state” (роль, маршрут, критичные флаги);
   - мягкая фоновая валидация вместо жесткого спиннера на весь экран.
3. Точечная оптимизация повторной инициализации в App/Auth:
   - не делать лишних тяжелых запросов на каждом ре-активировании вкладки;
   - добавить TTL для проверок MFA/password flags.
4. Прогон mobile-сценария “переключил приложение -> вернулся”.

### Этап B. Полноценные права доступа и симуляция ролей (приоритет 1)
1. Ввести явные права модулей в backend (в user_permissions):
   - can_access_afm_internal
   - can_access_adminscale
   - can_access_crm
   - (дополнительно) can_manage_crm_integrations
2. Сделать единый ModuleGuard:
   - защищает и меню, и прямые URL.
3. Привести страницы к effectiveRole:
   - чтобы “View as role” реально менял вид и поведение во всех целевых разделах.
4. AFM Settings:
   - сделать матрицу не декоративной, а реально сохраняемой/применяемой.

### Этап C. CRM 100%: интеграции, бот, инструкции (приоритет 1)
1. Боты:
   - экран управления ботами: добавить/заменить/деактивировать;
   - выбор активного бота на клиента/воронку;
   - тест с подробной диагностикой (бот недоступен, чат не найден, нет прав, и т.д.).
2. Внешние CRM:
   - коннекторы-сценарии (минимум: универсальный webhook + готовые шаблоны полей для популярных CRM);
   - мастер “шаг 1-2-3” с проверкой payload в реальном времени.
3. Инструкции:
   - интерактивные, с авто-подстановкой endpoint, копированием команд, чеклистом “готово/не готово”.
4. Логи:
   - единая лента доставки и ошибок по интеграциям (понятная для не-тех пользователей).

### Этап D. CRM рабочее пространство “AFM Digital” (приоритет 2)
1. Добавить в модель данных внутренний workspace агентства (без костылей в UI).
2. Показать в селекторе CRM “AFM Digital” вместе с клиентами.
3. Ограничить доступ к внутреннему workspace по правам.

### Этап E. Онлайн и значимая активность пользователей (приоритет 2)
1. Новый админ-экран:
   - кто онлайн сейчас;
   - последняя активность;
   - текущая/последняя страница.
2. Расширить логирование значимых действий:
   - только полезные бизнес-события (изменение прав, клиентов, задач, интеграций, этапов CRM и т.д.).
3. Фильтры:
   - по пользователю, типу действия, периоду, клиенту/workspace.

### Этап F. Переводы и финальная полировка (приоритет 2)
1. Полный аудит строк в измененных разделах.
2. Перенос всех текстов в i18n-слой.
3. Проверка RU/EN/IT/ES/AR/FR на ключевых сценариях.

## 5) Порядок выполнения и контроль результата

Порядок:
A (стабильность вкладок) -> B (доступы и role simulation) -> C (CRM интеграции и боты) -> D (AFM Digital workspace) -> E (онлайн/активность) -> F (переводы).

Критерии приемки:
1. Переключение вкладок/приложений не вызывает “полный перезапуск” интерфейса в типовом сценарии.
2. Неадмин не может открыть AFM Internal/AdminScale/CRM без выданного права даже по прямому URL.
3. В CRM можно выбрать конкретного бота, заменить его, протестировать и увидеть понятную диагностику.
4. В CRM доступен workspace “AFM Digital”.
5. Админ видит живой список онлайн + полезную историю действий по каждому аккаунту.
6. Ключевые экраны полностью локализованы на все поддерживаемые языки.
