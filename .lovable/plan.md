

# Полный аудит платформы AFM DIGITAL

---

## 1. АРХИТЕКТУРА И СТРУКТУРА

### Что готово и работает хорошо:
- **Ролевая система** — AgencyAdmin, MediaBuyer, Manager, Client с RLS-защитой на уровне БД
- **Модульная архитектура** — AFM Internal, AdminScale Pro, CRM изолированы с ModuleGuard
- **MFA** — двухфакторная аутентификация через TOTP (аутентификатор)
- **Dual Admin Approval** — критичные операции (удаление клиента, смена роли) требуют одобрения второго админа
- **Симуляция ролей** — "View as role" для проверки отображения интерфейса

### Проблемы:
- **ClientDetailPage.tsx — 1683 строки.** Это монолит. Содержит GoogleSheetConnection, CampaignsBreakdownTab и основную страницу в одном файле. Любое изменение рискует сломать соседнюю функциональность.
- **Дублирование логики** — ClientDashboardPage (666 строк) повторяет ~60% логики ClientDetailPage (графики, KPI, таблицы). Изменения нужно делать в двух местах.

## 2. ДАННЫЕ И СИНХРОНИЗАЦИЯ

### Что работает:
- Meta Ads API прямое подключение с ежечасным cron (72-часовое окно атрибуции)
- Google Sheets sync с автоматическим пропуском при наличии прямого Meta API
- Upsert-логика предотвращает дубликаты
- Тумблеры вкл/выкл для Meta Auto-sync и Google Sheets tracking

### Проблемы и нестыковки:
- **SyncMonitorPage показывает только platform_connections.** Он не показывает ad_accounts (прямые Meta подключения). Если клиент подключен через API без Google Sheets, Sync Monitor его не видит вообще.
- **Нет лога ошибок синхронизации в UI.** Таблица raw_api_logs существует, но нигде не отображается для диагностики.
- **Нет уведомлений при сбое синхронизации.** Если Meta API возвращает ошибку (истёкший токен, rate limit), администратор узнает об этом только зайдя в Sync Monitor или вручную проверив данные.
- **Budget progress bar привязан к текущему месяцу,** но DateRangePicker может показывать данные за другой период — это путает. Прогресс может показывать 80%, а данные в таблице — за 3 дня.

## 3. ВЕБХУКИ — ДВЕ НЕСВЯЗАННЫЕ СИСТЕМЫ

### Текущее состояние:
1. **Client Webhooks** (вкладка Webhooks в ClientDetailPage) — **исходящие.** Отправляют данные из платформы наружу (в CRM клиента, Telegram). Таблица: `client_webhooks`. События: `lead.new`, `metric.synced`, `report.published` и др.

2. **CRM Webhooks** (раздел CRM → Webhooks) — **входящие.** Принимают лиды из внешних источников (Meta Ads, лендинги) в CRM. Таблица: `crm_webhook_endpoints`. Edge-функция: `crm-webhook`.

### Нестыковки:
- **Полностью разные таблицы, разные UI, разная логика.** Пользователь может не понимать разницу.
- **Нет перекрёстных ссылок.** Из карточки клиента нельзя перейти к CRM-вебхукам этого клиента, и наоборот.
- **Client Webhooks (исходящие) реально не триггерятся.** Edge-функция `trigger-webhooks` существует, но нет гарантии что она вызывается из всех нужных мест (при создании лида, синхронизации метрик, публикации отчёта).

### Рекомендация:
Объединить интерфейс: на вкладке Webhooks клиента показать две секции — "Входящие (Сбор лидов)" и "Исходящие (Уведомления)". Добавить навигационную связь с CRM.

## 4. CRM

### Что работает:
- Канбан-доска с drag-and-drop
- Табличное представление лидов
- Настраиваемые стадии воронки (Qualified, Booked, Won, Lost)
- Webhook ingestion для автоматического сбора лидов
- Аналитика (ROAS, CPQL, Cost per Sale)
- Интеграции (Telegram бот, инструкции для внешних CRM)

### Проблемы:
- **CRM не связан с Campaign data.** Лиды в CRM и leads в daily_metrics/ad_level_metrics — это два полностью отдельных потока данных. Нет сквозной аналитики "кампания X → лид Y → продажа Z".
- **UTM-атрибуция не реализована.** В памяти упоминается "атрибуция по UTM и кампаниям", но в коде crm-webhook Edge-функция не парсит UTM-параметры и не привязывает лиды к конкретным кампаниям/объявлениям.
- **CRM Pipeline работает только с active клиентами.** `SELECT ... WHERE status = 'active'` — паузированные клиенты не видны в CRM, но их лиды могут продолжать приходить.

## 5. ОТЧЁТЫ

### Что работает:
- Создание отчётов с выбором секций (KPI, графики, таблицы)
- Привязка к клиенту и периоду
- Статусы draft/published

### Проблемы:
- **PDF-генерация не реализована.** Поле pdf_url существует в таблице, но нет Edge-функции для генерации PDF. Кнопка Download не работает.
- **Отчёты не отправляются клиентам.** Scheduled Reports таблица существует, но автоматической рассылки нет.
- **Нет preview отчёта.** После создания отчёт просто лежит записью в БД, но визуально не рендерится.

---

## 6. ЧАТ

### Что работает:
- Комнаты чатов (support, team, direct)
- Автоматическое создание support-комнаты при создании клиента
- Уведомления админам о сообщениях клиентов
- Загрузка изображений

### Проблемы:
- **Нет real-time обновлений.** ChatPage использует polling/manual refresh, не Supabase Realtime. Сообщения появляются только после перезагрузки или отправки.
- **Нет индикатора набора текста (typing indicator).**
- **Нет уведомлений о непрочитанных сообщениях** в самом чате (только в сайдбаре через notifications).

## 7. ЗАДАЧИ (Task Board)

### Что работает:
- Канбан-доска с drag-and-drop (pending → in_progress → completed)
- Фильтрация по клиентам
- Множественное назначение через task_assignees
- Привязка к клиенту

### Проблемы:
- **Task Reminders Edge-функция существует, но не подключена к cron.** Напоминания не отправляются автоматически.
- **Задачи из ClientDetailPage и TaskBoardPage — одни и те же данные,** но UI разный. В ClientDetailPage нельзя перетаскивать задачи, а в TaskBoardPage нет привязки к дедлайну в визуальном виде.
- **Нет приоритетов задач** (urgent, high, medium, low).

---

## 8. КЛИЕНТСКИЙ ДАШБОРД (Client Role)

### Что работает:
- Авто-определение клиента по client_users
- KPI карточки, график, daily table
- Воронка конверсии
- Просмотр опубликованных отчётов

### Проблемы:
- **Клиент не видит свои задачи** — TaskBoardPage недоступен для роли Client, хотя задачи могут быть назначены на клиента.
- **Нет раздела "Рекомендации"** или комментариев медиабайера — клиент видит только сухие цифры.
- **Budget bar для клиента** — клиент видит "сколько потрачено из бюджета", но не может посмотреть историю расходов по месяцам.

## 9. UI/UX ПРОБЛЕМЫ

- **Вкладка "Spend by Platform" на Overview клиента** проверяет наличие Google Sheets URL, но не проверяет наличие прямого Meta API подключения. Может показывать "Not Connected" даже если данные тянутся напрямую.
- **Календарь** — минимальная функциональность. Показывает задачи и аннотации, но нельзя создать событие из календаря.
- **Glossary** — существует, но нет данных в коде о том, откуда берутся определения. Скорее всего пустой.
- **Budget Planner** — отдельная страница, но budget plan также виден на ClientDetailPage. Неясна навигационная связь.
- **"Last Updated" на Overview** показывает `new Date().toLocaleDateString()` — то есть всегда сегодня. Это должно показывать реальное время последней синхронизации.

## 10. БЕЗОПАСНОСТЬ

### Что хорошо:
- RLS на всех таблицах
- SECURITY DEFINER функции для проверки ролей
- Vault для хранения токенов
- HMAC-подписи для вебхуков
- Audit Log

### Нуждается во внимании:
- **crm-webhook verify_jwt = false** — это корректно (внешний приём данных), но нет rate limiting.
- **approve-user verify_jwt = false** — это менее очевидно, нужно проверить, есть ли внутренняя валидация.
